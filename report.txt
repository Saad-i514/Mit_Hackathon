1. Executive Summary
This document serves as the complete technical specification and developer reference for The AI Scientist — an AI-powered, end-to-end experiment planning platform developed for the Fulcrum Science × Hack-Nation Global AI Hackathon 2026. The system accepts a natural-language scientific hypothesis and autonomously generates a fully operational experiment plan including protocol steps, materials with catalog numbers, realistic budget estimates, phased timelines, and success/failure validation criteria.
The platform is built to production-grade standards — not a demo or prototype. Every component uses battle-tested technologies with real API integrations, persistent storage, and a polished user interface that a practicing Principal Investigator would confidently use.
The implementation additionally fulfils the Stretch Goal: a Scientist Review & Learning Loop that captures structured expert corrections and uses them as few-shot context for subsequent plan generations of similar experiment types — creating a system that improves with use.

2. Project Overview
2.1  Problem Statement
Converting a scientific question into a runnable experiment plan currently requires weeks of manual expert labour: scoping the methodology, estimating costs, sourcing reagents, and staffing the team. A senior scientist who has run a similar protocol before may do this in hours; one who has not may take days — and the quality gap is real. A miscalculated reagent concentration or an unrealistic timeline can derail a lab for weeks.

2.3  Quality Bar
Standard:  Would a real scientist trust this plan enough to order the materials and start running it on Monday? That is the only acceptable quality threshold.

2.4  Sample Inputs Supported
• Diagnostics — paper-based electrochemical biosensor for C-reactive protein detection in whole blood
• Gut Health — Lactobacillus rhamnosus GG supplementation effect on intestinal permeability in C57BL/6 mice
• Cell Biology — trehalose vs DMSO cryoprotectant comparison for HeLa cell post-thaw viability
• Climate — Sporomusa ovata CO2 fixation to acetate in bioelectrochemical systems


3. Tech Stack — Selection & Justification
3.1  Why LangGraph (not CrewAI)
CrewAI is designed for autonomous multi-agent systems where agents decide their own sub-tasks. The AI Scientist pipeline is structured and sequential — a directed graph of well-defined nodes. LangGraph is purpose-built for exactly this pattern: it models the pipeline as a stateful directed graph with explicit transitions, making every step deterministic, traceable, and debuggable in LangSmith.

CREATE TABLE experiments (    id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),    hypothesis    TEXT NOT NULL,    domain        TEXT,                     -- e.g. 'diagnostics', 'cell_biology'    status        TEXT DEFAULT 'pending',   -- pending | qc_done | plan_done | reviewed    created_at    TIMESTAMPTZ DEFAULT NOW(),    updated_at    TIMESTAMPTZ DEFAULT NOW());
6.3  Table: literature_qc_results
CREATE TABLE literature_qc_results (    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),    experiment_id   UUID REFERENCES experiments(id) ON DELETE CASCADE,    novelty_signal  TEXT NOT NULL,   -- 'not_found' | 'similar_exists' | 'exact_match'    references      JSONB,           -- [{title, authors, year, url, abstract}]    raw_query       TEXT,    created_at      TIMESTAMPTZ DEFAULT NOW());
6.4  Table: experiment_plans
CREATE TABLE experiment_plans (    id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),    experiment_id    UUID REFERENCES experiments(id) ON DELETE CASCADE,    protocol         JSONB,    -- [{step_number, title, description, duration_hours}]    materials        JSONB,    -- [{name, catalog_no, supplier, unit, qty, unit_cost_usd}]    budget_total_usd NUMERIC,    budget_breakdown JSONB,    timeline         JSONB,    -- [{phase, week_start, week_end, tasks, dependencies}]    validation       JSONB,    -- {success_criteria, failure_criteria, controls, assays}    summary          TEXT,    model_used       TEXT,    tokens_used      INTEGER,    created_at       TIMESTAMPTZ DEFAULT NOW());
6.5  Table: scientist_reviews  (Stretch Goal)
CREATE TABLE scientist_reviews (    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),    plan_id         UUID REFERENCES experiment_plans(id) ON DELETE CASCADE,    experiment_id   UUID REFERENCES experiments(id) ON DELETE CASCADE,    section         TEXT NOT NULL,  -- 'protocol' | 'materials' | 'budget' | 'timeline' | 'validation'    rating          INTEGER CHECK (rating BETWEEN 1 AND 5),    correction      TEXT,           -- Free-text correction from scientist    is_critical     BOOLEAN DEFAULT FALSE,    reviewer_role   TEXT,           -- 'PI' | 'postdoc' | 'technician'    created_at      TIMESTAMPTZ DEFAULT NOW());
6.6  Table: feedback_store  (Stretch Goal — Few-Shot Context)
CREATE TABLE feedback_store (    id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),    experiment_type  TEXT NOT NULL,  -- domain/experiment category    section          TEXT NOT NULL,    original_text    TEXT,    corrected_text   TEXT NOT NULL,    correction_note  TEXT,    embedding        vector(1536),   -- OpenAI text-embedding-3-small embedding    times_used       INTEGER DEFAULT 0,    created_at       TIMESTAMPTZ DEFAULT NOW());-- HNSW index for fast similarity searchCREATE INDEX feedback_embedding_idx    ON feedback_store USING hnsw (embedding vector_cosine_ops);

7.3  LangGraph Pipeline Compilation
# app/graph/pipeline.pyfrom langgraph.graph import StateGraph, ENDfrom app.graph.state import ExperimentStatefrom app.graph.nodes.validate_input    import validate_input_nodefrom app.graph.nodes.literature_qc    import literature_qc_nodefrom app.graph.nodes.feedback_injector import feedback_injector_nodefrom app.graph.nodes.plan_generator   import plan_generator_nodedef build_pipeline() -> StateGraph:    graph = StateGraph(ExperimentState)    # Register nodes    graph.add_node("validate_input",     validate_input_node)    graph.add_node("literature_qc",      literature_qc_node)    graph.add_node("feedback_injector",  feedback_injector_node)   # ★ Stretch    graph.add_node("plan_generator",     plan_generator_node)    # Define edges    graph.set_entry_point("validate_input")    graph.add_edge("validate_input",    "literature_qc")    graph.add_edge("literature_qc",     "feedback_injector")       # ★ Stretch    graph.add_edge("feedback_injector", "plan_generator")          # ★ Stretch    graph.add_edge("plan_generator",    END)    # Conditional edge: if validation fails, go straight to END    graph.add_conditional_edges(        "validate_input",        lambda s: "literature_qc" if not s.get("error") else END,    )    return graph.compile()pipeline = build_pipeline()
7.4  Node: literature_qc
# app/graph/nodes/literature_qc.pyimport httpx, jsonfrom app.graph.state import ExperimentStatefrom app.config import settingsS2_BASE  = "https://api.semanticscholar.org/graph/v1"HEADERS_S2 = {"x-api-key": settings.SEMANTIC_SCHOLAR_API_KEY}async def literature_qc_node(state: ExperimentState) -> ExperimentState:    hypothesis = state["hypothesis"]    # ── 1. Semantic Scholar search ────────────────────────────────────────    async with httpx.AsyncClient(timeout=15) as client:        resp = await client.get(            f"{S2_BASE}/paper/search",            params={                "query":  hypothesis[:200],                "fields": "title,authors,year,abstract,externalIds,url",                "limit":  5,            },            headers=HEADERS_S2,        )    papers = resp.json().get("data", [])    # ── 2. Classify novelty ───────────────────────────────────────────────    # GPT-4o judges similarity using abstracts    from app.services.openai_client import classify_novelty    novelty, refs = await classify_novelty(hypothesis, papers)    # ── 3. Update state ───────────────────────────────────────────────────    return {        **state,        "novelty_signal": novelty,        "qc_references":  refs,    }
7.5  Node: plan_generator
# app/graph/nodes/plan_generator.pyimport jsonfrom openai import AsyncOpenAIfrom app.graph.state import ExperimentStatefrom app.config import settingsclient = AsyncOpenAI(api_key=settings.OPENAI_API_KEY)PLAN_SYSTEM_PROMPT = open("app/prompts/plan_generator.txt").read()async def plan_generator_node(state: ExperimentState) -> ExperimentState:    hypothesis       = state["hypothesis"]    novelty_signal   = state.get("novelty_signal", "not_found")    qc_refs          = state.get("qc_references", [])    feedback_context = state.get("relevant_feedback", [])   # ★ Stretch    # Build few-shot block from scientist corrections ★    few_shot_block = ""    if feedback_context:        few_shot_block = "\n\nRELEVANT EXPERT CORRECTIONS FROM PRIOR PLANS:\n"        for fb in feedback_context[:3]:            few_shot_block += f"Section: {fb['section']}\n"            few_shot_block += f"Correction: {fb['corrected_text']}\n---\n"    user_message = f"""Hypothesis: {hypothesis}Literature QC Signal: {novelty_signal}Prior References: {json.dumps(qc_refs, indent=2)}{few_shot_block}Generate a complete, operationally realistic experiment plan as JSON.Respond ONLY with valid JSON matching the schema below:{{  "protocol": [...],  "materials": [...],  "budget": {{}},  "timeline": [...],  "validation": {{}},  "summary": "..."}}"""    response = await client.chat.completions.create(        model="gpt-4o",        messages=[            {"role": "system", "content": PLAN_SYSTEM_PROMPT},            {"role": "user",   "content": user_message},        ],        response_format={"type": "json_object"},        max_tokens=4096,        temperature=0.2,    )    plan = json.loads(response.choices[0].message.content)    return {**state, **plan}

11. Stretch Goal — Scientist Review & Learning Loop
Stretch Goal Status:  INCLUDED IN FULL SCOPE. This is the highest-value differentiator — it transforms the tool from a one-shot generator into a platform that compounds in quality with every expert review.

11.1  Architecture Overview
Flow:  Scientist views generated plan       │       ▼  Scientist uses Review Panel (section-by-section)       │  Rating (1-5) + Free-text correction + Is-critical flag       │       ▼  POST /api/v1/reviews  →  saved to scientist_reviews table       │       ▼  Background task: embed correction using OpenAI text-embedding-3-small       │  → stored in feedback_store with experiment domain tag       │       ▼  Next plan generation for similar domain:  feedback_injector node queries pgvector for top-3 matching corrections       │  cosine similarity search on hypothesis embedding       │       ▼  Corrections injected as few-shot context into GPT-4o plan_generator prompt       │       ▼  New plan reflects expert corrections without re-prompting
11.2  Feedback Injector Node
# app/graph/nodes/feedback_injector.pyfrom app.services.embedding import get_embeddingfrom app.services.supabase_client import supabasefrom app.graph.state import ExperimentStateasync def feedback_injector_node(state: ExperimentState) -> ExperimentState:    hypothesis = state["hypothesis"]    # Embed the current hypothesis    embedding = await get_embedding(hypothesis)    # Query pgvector for top-3 most similar past corrections    result = supabase.rpc("match_feedback", {        "query_embedding": embedding,        "match_threshold": 0.75,        "match_count": 3    }).execute()    feedback = result.data or []    # Increment usage counter for retrieved entries    for fb in feedback:        supabase.table("feedback_store") \            .update({"times_used": fb["times_used"] + 1}) \            .eq("id", fb["id"]).execute()    return {**state, "relevant_feedback": feedback}
11.3  Supabase RPC Function for pgvector Similarity Search
-- Run in Supabase SQL EditorCREATE OR REPLACE FUNCTION match_feedback(    query_embedding vector(1536),    match_threshold float DEFAULT 0.75,    match_count     int   DEFAULT 3)RETURNS TABLE (    id             uuid,    section        text,    corrected_text text,    correction_note text,    times_used     int,    similarity     float)LANGUAGE sql STABLE AS $$    SELECT        id, section, corrected_text, correction_note, times_used,        1 - (embedding <=> query_embedding) AS similarity    FROM feedback_store    WHERE 1 - (embedding <=> query_embedding) > match_threshold    ORDER BY embedding <=> query_embedding    LIMIT match_count;$$;

15. Security Considerations
Risk
Mitigation
API key exposure in frontend
Only NEXT_PUBLIC_ prefixed env vars (Supabase anon key) go to the client. OpenAI, Serper, S2 keys stay server-side only in FastAPI.
Supabase anon key abuse
Row Level Security enabled on all tables. Anon key can only read/insert — not delete or update arbitrary rows.
Prompt injection via hypothesis input
Input is sanitised (stripped to 2000 chars), validated by Pydantic, and passed as a user message (not system message). No user input enters the system prompt.
OpenAI cost overrun
max_tokens=4096 enforced per request. Rate limiting middleware added to FastAPI (slowapi library — 10 req/min per IP).
CORS
CORS_ORIGINS env var restricts allowed origins — only Vercel domain and localhost in development.
Sensitive data in logs
LangSmith traces are project-scoped and require API key to view. API keys are never logged in application logs.

17. Glossary
Term
Definition
LangGraph
An open-source orchestration library by LangChain that models AI pipelines as stateful directed graphs with explicit node transitions.
LangSmith
Observability platform for LangChain/LangGraph applications. Provides full execution traces, token counts, and latency metrics.
SSE
Server-Sent Events — a one-directional HTTP streaming protocol used to push real-time pipeline progress events to the browser.
pgvector
A PostgreSQL extension that enables vector similarity search. Used here to find similar past scientist corrections for few-shot injection.
HNSW
Hierarchical Navigable Small World — an approximate nearest-neighbour index algorithm used by pgvector for fast similarity search at scale.
Few-shot context
Providing examples of correct outputs within the LLM prompt so that the model generates outputs that match the demonstrated pattern.
Novelty signal
The output of the Literature QC node: one of 'not_found', 'similar_exists', or 'exact_match' — indicating how novel the hypothesis is relative to published work.
CRO
Contract Research Organisation — a specialist lab that receives experimental briefs and generates proposals; The AI Scientist automates this process.
PI
Principal Investigator — the lead scientist on a research project who would typically review and approve an experiment plan.
RAG
Retrieval-Augmented Generation — providing retrieved context (e.g. prior feedback, literature) to an LLM alongside its generative prompt.
